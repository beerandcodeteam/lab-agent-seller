Um processo em 3 etapas

Além do contexto padronizado ao modelo, o harness estrutura o caminho da ideia ao código.

01 /init

Iniciar um projeto

Do zero ao MVP em poucas horas: descrição do projeto, user stories, schema de banco e fases prontas para execução.

02 /ai-context /plan

Evoluir um projeto existente

Documenta a realidade do código e transforma a descrição da feature em SPEC formal e plano faseado.

03 ./ralph.sh

Executar

As etapas 1 e 2 desembocam aqui: o plano de fases vira execução autônoma, validada a cada passo.

1 fase = 1 sessão

4 gates mecânicos

1 commit por fase

Como a LLM gera a resposta

Alto nível, sem purismo: um loop de próximo token.

Não existe "a resposta" guardada no modelo: existe o próximo passo mais provável dado o caminho já percorrido. O reasoning desenha a rota no rascunho antes de pisar.

O peso da persona

Nenhum token novo entra em cena: o campo inclina e a massa escorre.

Sem persona a resposta é genérica: a média de todo mundo que já escreveu sobre aquilo. A persona não dá bolinhas novas nem acerta mais, ela muda o peso das que já estavam lá.

Lost in the Middle

A atenção pesa nas pontas: instrução enterrada no meio é instrução ignorada.

Antídoto no harness: prompt enxuto e self-contained, com o crítico nas pontas: papel no início, template e self-checks no fim.

Context Rot

Sessão longa acumula lixo: instrução velha, tentativa falha, contradição.

Antídoto no harness: decisão vira artefato em disco, não memória de chat. E no ralph, cada fase nada em água limpa: sessão nova.

O método MALT

A maior dificuldade não é codar: é planejar antes de desenvolver.

O /init em 4 fases

O MALT aplicado: do alto nível ao faseamento, antes da primeira linha de código.

Muitos projetos morrem na fase 3, antes da primeira linha de código: errou na modelagem do banco, errou na vida do projeto.

O comando de descrição

A fase 1 do /init: onde a ideia vaga ganha contorno concreto.

O processo

/init:project-description
  1. 1

    Descobre o ambiente

    Stack com versões reais vem do projeto, não de chute do modelo.

  2. 2

    Entrevista pra fechar lacunas

    O que é o projeto, o que não pode ficar de fora, o que fica de fora, como o usuário lida com ele, quais as regras de negócio.

  3. 3

    Escreve o artefato

    Overview, conceitos-chave, tech stack e workflows numerados em .spec/init/project-description.md.

  4. 4

    Valida até ficar verde

    Self-checks mecânicos; falhou, corrige e roda de novo.

Os conceitos por trás do prompt

Comando é só um prompt? Sim: um prompt com engenharia. O project-description.md por dentro.

commands/init/project-description.md

---

allowed-tools: Read, Write, Edit, Bash,
  AskUserQuestion... 1

---

You are helping a developer turn a rough
idea into a structured description... 2

$ARGUMENTS 3

## Your goal (1 a 5) 4

## Process

### 1. Discover the environment first

### 2. Interview to close gaps

### 3. Write the document 5

### 4. Self-checks (run until green) 6

### 5. Close out

  • 1

    Menor privilégio. O cabeçalho declara só as ferramentas que o comando usa. Nada de acesso sobrando.

  • 2

    Papel e porquê. Uma frase define a persona e o peso do artefato: fonte de verdade, escrita antes do código.

  • 3

    Input isolado. A ideia do dev entra num bloco próprio, separada das instruções. O prompt prevê até input vazio.

  • 4

    Objetivos enumerados. Cinco metas verificáveis: o modelo sabe exatamente o que é "pronto".

  • 5

    Processo ordenado + template verbatim. Descobrir antes de perguntar, perguntar antes de escrever; e o output segue "exactly this structure", sem formato inventado.

  • 6

    Fecha com prova. Self-checks em bash dentro do próprio prompt e relatório final padronizado, vermelho até verde.

Regra transversal: toda afirmação rastreável ao dev ou ao ambiente. Lacuna vira Open Question, nunca invenção.

Como construir um bom comando

Todo prompt promete. Comando bom prova. Seis conceitos que viram precisão de entrega.

Self-checks mecânicos

O comando prova o próprio output com bash, vermelho até verde. Estrutura, contagens e formato são verificados, não prometidos.

Cobertura verificável

"Cobre tudo" só vale com enforcement: cada workflow vira story, cada story rastreada em task, via grep e diff. Faltou, o check acusa.

Re-run sem perda

Reexecutar atualiza só os deltas: IDs estáveis, nunca renumera, edição do dev respeitada. Regravar do zero destrói trabalho.

Detecção de drift

Cada artefato carimba o sha256 dos insumos. Insumo mudou, downstream acusa stale: avisa, nunca bloqueia.

Roteador fino

O roteador lê o estado da cadeia e invoca um comando por execução. Ele mesmo não escreve nada: quem gera conteúdo é o comando especialista.

Contrato de máquina

Se um script consome o formato (o ralph parseia ## Phase N:), isso é contrato documentado e validado, não coincidência.

Hooks: código no loop do agente

Prompt é sugestão. Hook é lei: script determinístico intercepta cada evento da sessão.

Como funciona

  1. 1

    Um evento dispara

    Sessão abre, prompt chega, ferramenta vai rodar, turno termina... cada momento do loop tem um evento.

  2. 2

    Seu script recebe JSON no stdin

    Quem é a ferramenta, qual o comando, qual o cwd. O script decide com lógica, não com probabilidade.

  3. 3

    A saída decide o destino

    Deixa passar, bloqueia devolvendo o motivo ao agente, ou injeta contexto novo na conversa.

exit 0

passa; stdout pode virar contexto

exit 2

bloqueia; stderr volta pro agente

JSON

decisão fina: allow / deny / ask

A configuração

Hook real deste harness — hooks/hooks.json do plugin:

{

"hooks": {

"PreToolUse": [{

"matcher": "Bash",

"hooks": [{

"type": "command",

"command": "sail-guard.sh"

}]

}]

}

}

  • matcher filtra o alvo: só chamadas de Bash, só Edit|Write, ou regex.
  • Onde mora: settings.json (usuário, projeto ou local) ou hooks.json de plugin.
  • Vários hooks no mesmo evento: rodam em paralelo; a decisão mais restritiva vence.

Hooks: o cheatsheet

Todos os eventos, agrupados por momento do loop. ★ = os que você vai usar primeiro.

Sessão

  • SessionStart ★injeta contexto ao abrir/retomar
  • SessionEndlimpeza ao encerrar
  • Setuppreparação em CI/scripts

Prompt

  • UserPromptSubmit ★valida/enriquece antes do modelo ver
  • UserPromptExpansionintercepta expansão de comando

Ferramentas

  • PreToolUse ★antes de rodar — pode bloquear
  • PostToolUse ★depois — formatar, logar, corrigir
  • PostToolUseFailurequando a ferramenta falha
  • PostToolBatchapós lote paralelo

Permissões

  • PermissionRequestauto-aprova o sabidamente seguro
  • PermissionDeniedreage a negações automáticas

Contexto

  • PreCompact / PostCompactre-injeta o crítico após compactar
  • InstructionsLoadedCLAUDE.md entrou no contexto
  • ConfigChange · FileChanged · CwdChangedreage a mudanças no ambiente

Fim de turno

  • Stop ★bloqueia o fim se a tarefa não terminou
  • Notification ★alerta no desktop quando precisa de você
  • StopFailure · MessageDisplayerro de API · filtra texto exibido

Subagentes & tasks

  • SubagentStart / SubagentStopmonitora e limita subagentes
  • TaskCreated / TaskCompletedaudita o quadro de tarefas

Worktree & MCP

  • WorktreeCreate / WorktreeRemovecustomiza isolamento git
  • Elicitation / ElicitationResultformulários de servidores MCP

Além de command, um hook pode ser http, mcp_tool, prompt ou agent — até um subagente inteiro validando o evento.

Na prática: o sail-guard

Um PreToolUse deste harness: o erro vira instrução antes de acontecer.

Sem o hook

$ php artisan migrate

SQLSTATE[HY000] connection refused

$ php artisan migrate // de novo

SQLSTATE[HY000] connection refused

$ php -v

php: command not found

Num projeto com Sail, PHP e banco só existem dentro do container. O agente tenta no host, falha, tenta variações... loop de erro queimando tokens.

Com o hook

BLOQUEADO pelo sail-guard: este projeto

usa Laravel Sail.

Comando "php artisan migrate" rodaria no

HOST — e vai falhar.

Use o equivalente via Sail:

./vendor/bin/sail artisan migrate

  • Detecta vendor/bin/sail subindo do cwd até a raiz.
  • exit 2: bloqueia a chamada; o stderr acima vira contexto do agente.
  • Corrige sozinho: a próxima chamada já vem com sail na frente. Zero humano no meio.

Guardrail bom não só nega: nega explicando o caminho certo. Bloqueio sem sugestão vira outro loop de tentativa e erro.

Laravel Boost

O ecossistema Laravel empacotou o harness de contexto: guidelines, skills e MCP prontos de fábrica.

Guidelines

Regras de código mantidas pelo próprio time do Laravel, específicas para cada pacote e cada versão que o seu projeto realmente usa.

Skills

Instaladas em .claude/skills/: pest-testing, livewire-development, laravel-best-practices... O agente ativa a skill do domínio em que está mexendo.

Servidor MCP

Ferramentas que rodam dentro da aplicação: schema real do banco, tinker, logs, docs da versão certa. O agente pergunta pra app, não chuta.

$ composer require laravel/boost --dev

$ php artisan boost:install

  • Dois comandos: guidelines no AGENTS.md, skills no projeto, MCP no .mcp.json.
  • O boost:update roda no post-update do composer: atualizou pacote, contexto acompanha.

Guidelines que sabem a sua versão

Caso real: o agent-seller, um SaaS multi-tenant com agente de vendas por IA.

Detectado do composer

PHP 8.5 · Laravel v13

Livewire v4 · Pest v4

Tailwind v4 · laravel/ai v0.9

Os templates vivem no vendor por versão: .ai/php/8.5/, .ai/livewire/4/... O boost monta só o que se aplica ao seu lock — nada de regra de Livewire 2 num projeto Livewire 4.

O padrão a copiar para qualquer stack: o agente alucina quando não tem onde olhar. Dê a ele schema real, log real e doc da versão certa — cada resposta nasce de evidência.

/ai-context: realidade, não intenção

O agente não pode confiar em documentação que mente. Código não mente.

O problema

  • Documentação escrita à mão nasce boa e apodrece em silêncio: o código evolui, o texto fica parado no tempo.
  • Agente que lê doc velha planeja sobre um sistema que não existe mais — e o erro só aparece no diff.
  • Spec descreve intenção. Intenção e realidade divergem no primeiro sprint.

A regra AS IS

  • Únicas fontes: código, manifests, CI e configs — o que está implementado, com evidência citável.
  • .spec/ é invisível: o comando nunca lê planejamento. Plano não é prova de nada.
  • Nunca inventa: afirmação sem evidência não entra; a seção declara N/A e pronto.

"Documenta a realidade, não a intenção" — por isso o /plan confia no que o /ai-context escreveu: é retrato do código, auditável linha a linha.

/plan: da descrição ao plano executável

Três agentes especialistas, um checkpoint humano entre cada um.

01 specifier

Especifica

Descrição confirmada vira SPEC.md formal: requisitos com ID, ACs binários, diagramas AS IS / TO BE. Ambiguidade vira marcador [NEEDS CLARIFICATION], nunca chute.

02 clarifier

Interroga

QA adversarial em 2 fases: analisa a SPEC e devolve perguntas priorizadas por risco de retrabalho; você responde; ele aplica as respostas na SPEC e incrementa a versão.

03 planner

Decompõe

PLAN.md com tasks e riscos + PHASES.md no formato que o ralph executa + contratos formais (OpenAPI, proto, AsyncAPI) quando a SPEC declara API.

Humano em todo checkpoint

Confirma o input normalizado, aprova a SPEC, responde as ambiguidades, assina a decomposição. O agente propõe; você decide. Escopo cresceu? O pipeline para e propõe dividir.

Gate de arquitetura

Antes de planejar, o pipeline exige o AGENTS.md / docs/agents/ do /ai-context. Sem eles, avisa e marca missing — nunca planeja em silêncio sem conhecer o código.

Esforço proporcional, fim executável

O tier vem de sinais objetivos, não de opinião — e tudo desemboca no ralph.

light

≤ 3 requisitos, um repo, sem contrato formal. SPEC enxuta, decomposição inline, sem burocracia desnecessária.

standard

4–10 requisitos. SPEC completa, plano faseado, contratos quando a SPEC declara superfície de API.

complete

11+ requisitos, multi-repo ou domínio pesado. Clarifier obrigatório, contrato para cada interface exposta.

  • Mesma máquina da aula: planejou, o ralph executa — uma fase por sessão, gates mecânicos, um commit por fase.

O arco completo do harness

Os 4 pontos do arnês do primeiro slide, agora preenchidos.

CONTEXTO

/ai-context retrata o código, o Boost traz o framework, o MCP responde com fato. O agente conhece o projeto de verdade.

COMANDOS

/init para nascer, /plan para evoluir: prompts com engenharia — papel, processo, template, self-checks.

VALIDAÇÃO

Hooks interceptam o loop, self-checks provam o output, gates do ralph verificam fora da sessão. Confiança é mecânica, não promessa.

LOOPS

ralph.sh executa fase a fase em sessão fresca, corrige com causa real, commita só o validado. O humano fica nos checkpoints.
